# Coding harness

By default the agent can *talk* about code but can't touch it — every change has to
be described in chat and applied by hand. The **coding harness** gives it a small set
of file tools so it can read, edit, search, and run commands on a real codebase
directly, like a built-in coding assistant.

It is deliberately minimal — 4 base tools (`read`, `write`, `edit`, `bash`). Out of
scope: AST/structural editing, language-server integration, a plugin system, and a
terminal/REPL. Listing and searching files goes through `bash` (`ls`, `find`, `rg`)
instead of dedicated tools.

## Tools

All file paths are relative to the configured workspace root (or absolute *inside* it).

| Tool | Approval | What it does |
| --- | --- | --- |
| `read(path, offset=0, limit=100)` | pre-approved | Read a file (or a line range). Each line comes back as `LINE#HH:content` — the `LINE#HH` prefix is a **hashline anchor** usable in `edit`. Large files paginate via `offset`/`limit`. |
| `write(path, content)` | **asks first** | Write a file, creating parent directories. Overwrites. |
| `edit(path, edits=[...])` | **asks first** | Apply one or more edits in a single call, all-or-nothing (see below). |
| `bash(command, purpose, workdir=".")` | rule-gated | Run a shell command. Allowlisted commands behave like the old `run_command`; anything else runs **workspace-confined** and asks on every call. |

Reads are pre-approved (`ALWAYS`); `write`/`edit` are write-actions that ask for your
approval on every call, surfaced through the same [permission](/docs/permissions) flow
as email/calendar/message sends.

### Multi-edit and hashline anchors

`edit` takes an array of edits, validated together and applied atomically — nothing is
written unless every edit succeeds. Each edit is one of:

```json
{"path": "src/main.py", "edits": [
  {"oldText": "foo()", "newText": "bar()"},
  {"pos": "12#KT", "end": "15#BH", "lines": ["replacement", "lines"]}
]}
```

- `{oldText, newText}` — exact-substring replace. `oldText` must be unique in the file
  unless `"all": true` replaces every occurrence.
- `{pos, end?, lines}` — replace the line (or inclusive range) addressed by a `LINE#HH`
  anchor from `read` with the given lines (`[]` deletes). The 2-char hash covers the
  line *and its neighbours*, so if the file changed since the read the anchor no longer
  matches and the edit is rejected instead of landing on the wrong line.

Anchors let the model make targeted changes without quoting whole code blocks back,
which keeps context usage down (inspired by
[pi-hashline-edit](https://github.com/RimuruW/pi-hashline-edit)).

## One `bash` tool, two rails

`bash` replaces the old `run_command` + `run_command_in_dir` pair:

- A command matching the **prefix allowlist** (`himalaya`, `gh`, `git`, `wacli`,
  `sqlite3`, `rg`, …) runs exactly as `run_command` always did: read-style commands are
  pre-approved by [permission rules](/docs/permissions) (`bash:git*log*` etc.), risky
  ones ask.
- Any **other** command (builds, tests, `make`, `npm`, …) requires the workspace: it
  runs with its working directory confined to the workspace root (or `workdir` inside
  it) and asks for approval on **every** call — approvals for these are never
  auto-learned into `ALWAYS` rules.

The hard `NEVER` rails apply to both rails. Note the confinement scope: the workspace
bounds the **working directory**, not what an approved command may touch — the command
runs with the full privileges of the humux process. Its real boundary is your approval.

## Confinement

The **file** tools (`read`/`write`/`edit`) are confined to the single **workspace
directory** you configure. The check is `realpath`-based: a path is resolved (following
symlinks and collapsing `..`) and must land on the workspace root or a descendant of
it. So neither `../../etc/passwd` nor a symlink pointing outside the workspace can
escape — both are blocked.

There is **no default workspace**. With the harness disabled, or with a blank
directory, the file tools are hidden from the agent entirely and the handlers refuse —
and non-allowlisted `bash` commands have nowhere to run.

## One shared root

When the harness is on, `bash` runs **in the workspace root**, the same tree the file
tools resolve against. So a repo cloned with `bash git clone <url>` lands inside the
workspace and is immediately visible to `read`, editable with `edit`, and committable
with `git` — no need to bridge two directories. The root is created on first use if it
doesn't exist yet.

## Per-agent home directory

The system prompt gives each agent a home directory named after its
[agent](/docs/agents) slug — e.g. an agent named `coach` clones and writes under
`coach/…`, the default identity under `default/…`. This keeps multiple agents from
colliding in the workspace root when they share one directory. The public
[artifacts](/docs/artifacts) directory follows the same convention
(`artifacts/<slug>/`).

## Configuration

Enable it under **Settings → Workspace** in the admin UI, or in `config.yml`:

```yaml
workspace:
  enabled: true
  directory: "~/projects/my-repo"   # the only directory the agent may touch
```

Config is applied live — the tool set is recomputed each turn, so a change takes effect
on the agent's next turn (no restart or `/new` needed). Each call is written to the agent
tool log on the **Logs** tab.

Like every tool, the harness can be scoped per [agent](/docs/agents) — grant it to
a "developer" agent and leave it off everywhere else.
